## Analysis

**What worked:**
- LoRA ratio 2.0 (vs 0.25) made SFT converge much better
- 5 epochs (vs 2) gave lower loss
- packing=False + weight_decay=0.01 + train_on_responses_only = clean training
- Format gate diagnostic (with 2048 tokens) confirmed model learned format during SFT

**What didn't work:**
- GRPO is broken with Unsloth 2026.2.1 (3 tensor size bugs + KL instability)
- merged_4bit_forced may be destroying `<think>` capability
- Model outputs bare column names at inference despite `col:val` in training data
- System prompt at inference may not match training format exactly

**Why no `<think>` at inference?**
Likely causes (ranked by probability):
1. `merged_4bit_forced` rounding errors destroy fine-tuned behavior
2. System prompt mismatch between training and inference
3. Need explicit "Think step by step" in the user prompt
4. 5 epochs insufficient for `<think>` to become default behavior

## LLM Council Diagnosis (2026-03-07)

**Council:** Sonnet 4.6 (#1), Grok 4.1 (#2), Gemini 3 Flash (#3), DeepSeek v3.2 (#4)  
**Chairman:** Claude Opus 4.6  
**Full report:** `docs/reports/council_v2_eval_2026-03-07.json`

**UNANIMOUS ROOT CAUSE:** `merged_4bit_forced` destroys LoRA deltas via double quantization error.
- SFT training SUCCEEDED (format gate 3/3 before merge)
- Merge dequantizes → adds LoRA → re-quantizes 4-bit → LoRA signal rounded to zero
- Format behaviors (`<think>`, `col:val`) are small-magnitude probability shifts that 4-bit re-quantization erases

**Council recommendations:**
1. Run adapter-only eval (no merge) — expected F1 > 40% if merge was the only problem
2. NEVER use `merged_4bit_forced` for behavioral fine-tuning
3. Deploy via LoRA adapters separate OR 16-bit merge → GPTQ/AWQ
4. Update system prompt to explicitly mention col:val format
5. Increase dataset to 1000+ examples with explicit algorithmic CoT
6. Skip GRPO for v3, use rejection sampling for v4
